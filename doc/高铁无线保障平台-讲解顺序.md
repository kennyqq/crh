# 高铁无线保障平台讲解顺序

## 1. 讲解总原则

- 页面本身按真实系统设计，不在界面上展示讲解词、分镜、话术提示。
- 讲解时只围绕页面上已高亮的核心对象展开，避免频繁切换和无关操作。
- 整体故事线固定为：
  - 先看线路态势
  - 再看问题洞察
  - 再看策略保障
  - 最后看投诉回溯

---

## 2. 推荐讲解路径

### 第一步：进入“线路态势”

页面目标：

- 先建立“全网可视、线路可查、列车可跟”的第一印象。

讲解顺序：

1. 先看顶部模块栏，说明这是统一的高铁无线保障平台。
2. 视线落到左侧线路态势图，说明平台可以统一查看线路、站点、列车与重点区段。
3. 再看右上角列车详情卡，说明当前系统已经锁定一趟重点列车。
4. 最后看右下“重点区段清单”和左下“体验对比曲线”。

建议点击点：

- 当前带呼吸高亮的重点区段
- 当前高亮车次

建议讲法：

- 这一步只强调“能看见”，不要马上讲太多根因。
- 重点讲两件事：
  - PRS 口径和业务感知口径是有差异的
  - 不同车型在同一区段上会呈现不同的信号特征

---

### 第二步：切到“感知洞察”

页面目标：

- 证明平台不只是看见问题，还能把问题说清楚。

讲解顺序：

1. 先看左侧“问题列表”，说明系统会把高铁常见问题按模板化方式收敛。
2. 点击当前高亮的问题卡。
3. 看中间地图，说明问题区段已经被同步定位。
4. 再看右下“根因分析”和左下“业务指标”。

建议点击点：

- 当前呼吸高亮的问题卡

建议讲法：

- 这一步只讲“为什么会这样”。
- 要把“现象”和“原因”分开：
  - 现象是 RTT 上升、速率下降、驻留异常、质差次数增加
  - 原因是弱覆盖、隧道、700M 小带宽、5G 掉 4G 等

---

### 第三步：切到“策略保障”

页面目标：

- 证明系统不是只会看，还能根据业务和用户层级做策略干预。

讲解顺序：

1. 先看上方“保障模板”，说明平台支持按业务类型调用不同保障方案。
2. 点击当前高亮的业务场景。
3. 看左下“策略参数”，说明 5QI、RFSP、调度优先级如何变化。
4. 再点“应用保障策略”，看右下“保障前后对比”。

建议点击点：

- 当前高亮的保障模板
- “应用保障策略”按钮

建议讲法：

- 这里不要讲底层算法，重点讲“策略能解释、效果能验证”。
- 可以直接用结果说话：
  - 抖音卡顿下降
  - 游戏时延改善
  - 微信上传提升
  - 下载速率提升

---

### 第四步：切到“投诉回溯”

页面目标：

- 用一个投诉案例收尾，把前面的能力闭环起来。

讲解顺序：

1. 先看左侧投诉列表，说明系统支持匿名化投诉查询。
2. 点击当前高亮投诉。
3. 看中间地图和曲线，说明可以把投诉时段回放成 5 秒级证据链。
4. 再看右侧“问题分析”和“闭环建议”。

建议点击点：

- 当前高亮投诉
- 中间时间轴滑块

建议讲法：

- 这一页要讲“从投诉反查网络”。
- 把故事讲清楚：
  - 用户什么时候投诉
  - 当时列车在什么位置
  - 当时网络指标发生了什么
  - 系统判断最可能原因是什么
  - 后续应该怎么闭环

---

## 3. 推荐话术骨架

如果需要 2 到 3 分钟快速讲解，可以按下面顺序：

1. 先用“线路态势”说明平台可以统一看全网线路、重点列车和风险区段。
2. 再用“感知洞察”说明平台不只是看到异常，还能判断问题类型和根因。
3. 再用“策略保障”说明平台可以按业务、按用户做差异化保障，并验证效果。
4. 最后用“投诉回溯”说明平台可以从投诉记录快速回放证据链，形成闭环建议。

---

## 4. 注意事项

- 不要在讲解时强调“这是 Demo”，按真实系统口径讲。
- 不要讲高德地图在线或离线切换。
- 不要讲讲解员控制台，因为界面里已经去掉了。
- 每一页只抓一个高亮点，避免信息过载。
- 如果现场时间短，优先讲：
  - 线路态势
  - 感知洞察
  - 投诉回溯
